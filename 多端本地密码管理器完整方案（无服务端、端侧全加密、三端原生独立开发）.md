# 多端本地密码管理器技术规格

> 版本：1.0（设计基线）  
> 适用平台：Windows、macOS、Android  
> 产品定位：离线密码库 + 加密备份/迁移，不提供云端或实时同步服务。

## 1. 目标与关键决策

本产品在 Windows、macOS、Android 上提供原生密码管理应用。所有密码库数据仅保存在用户设备上；设备之间通过用户主动传输的加密 `.pwdlock` 文件迁移或备份数据。

以下决策为不可变的设计基线：

1. 不建设服务端、不登录账号、不上传用户数据，也不实现局域网自动发现或实时 Socket 同步。
2. 每个密码库使用随机生成的 **Vault Key** 加密；用户主密码不直接作为数据库或导出文件的数据密钥。
3. 主密码通过 Argon2id 派生密钥加密（包装）Vault Key；导出文件使用独立的导出密码和独立密钥。
4. 使用 AES-256-GCM 提供机密性与完整性；禁止自行实现 AES、GCM、Argon2id 或随机数生成器。
5. `.pwdlock` 是公开可解析的二进制容器规范，但其内容必须只能由持有正确导出密码的人解密。文件扩展名和魔数不构成安全性。
6. 文件导入不是“最后写入者获胜”的同步。若同一记录出现不同内容，必须保留两个版本并要求用户处理，绝不静默覆盖。

## 2. 范围与非目标

### 2.1 本期范围

- 主密码创建、解锁、修改、自动锁定；
- 加密本地密码库；
- 登录项的创建、查看、复制、编辑、删除、搜索和分类；
- 加密 `.pwdlock` 文件导出、导入、冲突处理与备份恢复；
- 强密码生成、弱密码/重复密码提示；
- 系统生物识别快捷解锁（可选能力，受平台支持限制）；
- 三端使用同一份 `.pwdlock` 文件协议和测试向量。

### 2.2 明确非目标

- 云同步、多人共享、账户登录、远程恢复和客服找回主密码；
- 自动合并所有并发修改；
- 浏览器自动填充、浏览器扩展、TOTP、附件、支付卡等高级数据类型（可作为后续独立版本）；
- 对已解锁设备上的恶意软件、键盘记录器、root/jailbreak 或系统级内存读取提供绝对防护；
- 将“禁止截图/录屏”作为跨平台可保证的安全承诺。

主密码遗失后，密码库不可恢复；首次设置和每次修改主密码时必须明确提示这一点。

## 3. 威胁模型与安全边界

### 3.1 需要防护的场景

- 设备丢失、硬盘被复制、应用数据库或 `.pwdlock` 文件泄露；
- 传输途中的文件被替换、截断或随机篡改；
- 普通文本编辑器、压缩工具或不掌握密码的第三方程序读取导出文件；
- 设备借用期间，应用处于锁定状态。

### 3.2 不承诺防护的场景

- 应用已解锁时，恶意软件、键盘记录器、辅助功能滥用或系统管理员读取界面、剪贴板、内存；
- 用户在钓鱼界面、远程控制软件或已失陷设备中输入主密码；
- 设备 root/jailbreak、内核级攻击、被植入的操作系统；
- Windows/macOS 的系统级截屏、录屏或摄像头拍摄屏幕。

应用私有目录、应用沙盒、自动锁定和防截屏均是降低风险的附加措施，不能替代加密或强主密码。

## 4. 术语

| 术语 | 含义 |
|---|---|
| Vault | 一个独立的本地密码库。 |
| Vault Key | 随机生成的 32 字节数据密钥，用于打开本地加密数据库。 |
| KEK | 由主密码或导出密码派生的 Key Encryption Key，仅用于包装随机数据密钥。 |
| 包装（wrap） | 以 AES-256-GCM 加密一个 32 字节数据密钥。 |
| `.pwdlock` | 可跨端导入的加密快照文件。 |
| 冲突组 | 同一逻辑记录 ID 出现不同内容时，保留下来的待用户裁决版本集合。 |
| 墓碑（tombstone） | 表示记录已删除的同步元数据；用于防止旧快照无意恢复已删记录。 |

## 5. 总体架构

```text
主密码 ──Argon2id──> 主密码 KEK ──AES-GCM 包装──> Vault Key ──> SQLCipher 数据库

导出密码 ──Argon2id──> 导出 KEK ──AES-GCM 包装──> Export Key ──AES-GCM──> .pwdlock 载荷
```

### 5.1 为什么采用信封加密

Vault Key 与主密码分离后，修改主密码只需重新包装 Vault Key，不必重加密整个数据库。导出文件同样使用独立的 Export Key 和导出密码，因此泄露主密码不会自动等同于泄露某个已经独立设置密码的导出文件，反之亦然。

任何主密码、导出密码、KEK、Vault Key、Export Key 都不得写入日志、崩溃报告、分析事件或明文配置文件。

## 6. 密码学规范

### 6.1 通用规则

- 随机数必须来自平台加密安全随机源：Windows `RandomNumberGenerator`、Apple `SecRandomCopyBytes`、Android `SecureRandom`。
- 密码先按 Unicode NFC 规范化，再以 UTF-8 字节传入 KDF；三端必须使用完全一致的规范化实现。
- 所有整数使用无符号大端序（network byte order）。
- AES-GCM nonce 固定为 12 字节，认证标签固定为 16 字节。相同密钥下 nonce 绝不能重复。
- 解密失败只返回统一错误“密码错误或文件损坏”，不得向调用方区分认证失败、密码错误或具体解析位置。
- 禁止使用 ECB、CBC、固定 IV、裸 SHA-256 哈希密码，或将加密算法实现为自写代码。

### 6.2 主密码 KDF

主密码和导出密码均使用 Argon2id（RFC 9106 兼容实现）。初始参数如下，实际参数必须随文件保存而非由客户端猜测：

| 参数 | 初始值 |
|---|---:|
| 算法 | Argon2id |
| 盐 | 随机 16 字节 |
| 输出长度 | 32 字节 |
| 内存成本 `m` | 65,536 KiB（64 MiB） |
| 时间成本 `t` | 3 |
| 并行度 `p` | 1 |
| 版本 | Argon2 v1.3 |

新版本可以提高参数，但不得降低已有库的参数。若设备不能满足参数，应提示用户设备不受支持；不得静默降级。参数升级仅在用户成功解锁后、明确完成兼容性验证时执行。

### 6.3 Vault Key 包装

创建密码库时：

1. 生成随机 32 字节 Vault Key；
2. 生成随机 16 字节主密码盐；
3. 用 Argon2id 从主密码派生 32 字节主密码 KEK；
4. 生成随机 12 字节包装 nonce；
5. 使用 AES-256-GCM，以主密码 KEK 加密 Vault Key；
6. 将固定头与可变 KDF 参数作为 AAD；保存密文和认证标签。

应用验证主密码时，尝试认证解密 Vault Key。无需另存“主密码哈希”；后者既重复又会为离线猜测提供额外验证目标。

修改主密码时，必须在内存中先用旧密码解开 Vault Key，再用新主密码生成的新 KEK 重新包装该 Vault Key。数据库内容不重加密。

`vault.meta` 的 v1 格式为：`PVLT`（4 B）+ `metaVersion`（1 B，`0x01`）+ `vaultId`（16 B 原始 UUID）+ `kdfId`（1 B）+ `aeadId`（1 B）+ `flags`（1 B，必须为零）+ `memoryKiB`（4 B）+ `iterations`（4 B）+ `parallelism`（1 B）+ `reserved`（3 B，必须为零）+ `passwordSalt`（16 B）+ `wrapNonce`（12 B）+ `wrappedVaultKey`（32 B）+ `wrapTag`（16 B）。`wrappedVaultKey` 的 AAD 为从 `PVLT` 到 `passwordSalt` 的全部前置字节。文件无尾随字节；任一长度、版本、保留位或认证校验不符合即拒绝打开。

### 6.4 本地数据库

- Windows/macOS 使用 SQLite + SQLCipher；Android 使用 Room + SQLCipher。
- SQLCipher 数据库密钥为 Vault Key 的原始 32 字节值，使用库支持的原始二进制密钥接口传递，禁止把十六进制文本当作普通口令再次派生。
- SQLCipher 的具体版本、KDF/页大小/兼容参数必须在各端锁定并记录；数据库文件不作为跨端数据格式交换，跨端只支持 `.pwdlock`。
- 元数据文件 `vault.meta` 仅保存 Vault Key 包装信息、库 ID 和协议版本，不保存密码、KEK 或明文记录。
- 数据库与元数据必须位于应用私有目录；数据库写入、导入和主密码修改均采用临时文件 + fsync + 原子替换，保留最近一次可验证备份。

## 7. `.pwdlock` 文件协议

### 7.1 设计原则

`.pwdlock` 的二进制格式用于稳定互操作和安全解析，而非隐藏算法。协议字段必须版本化、限制长度并被认证；任何未知的关键字段或版本必须拒绝导入。

文件最大大小为 100 MiB；解密后的 JSON 最大 200 MiB；单次导入最多 100,000 条记录。超限时在分配大块内存前失败。

### 7.2 二进制外层（v1）

所有字段按顺序写入：

| 字段 | 大小 | 说明 |
|---|---:|---|
| magic | 4 B | ASCII `PWLK`（`0x50 57 4C 4B`） |
| formatVersion | 1 B | 固定为 `0x01` |
| kdfId | 1 B | `0x01` = Argon2id |
| aeadId | 1 B | `0x01` = AES-256-GCM |
| flags | 1 B | v1 必须为 `0x00`，非零拒绝导入 |
| memoryKiB | 4 B | Argon2id `m` |
| iterations | 4 B | Argon2id `t` |
| parallelism | 1 B | Argon2id `p` |
| reserved | 3 B | 必须全零 |
| passwordSalt | 16 B | 导出密码盐 |
| wrapNonce | 12 B | 包装 Export Key 的 nonce |
| wrappedExportKey | 32 B | 被导出 KEK 加密后的 Export Key 密文 |
| wrapTag | 16 B | 包装 Export Key 的 GCM 标签 |
| payloadNonce | 12 B | 加密载荷的 nonce |
| payloadLength | 8 B | 载荷密文长度，最大 100 MiB |
| payloadCiphertext | 可变 | 加密后的 UTF-8 JSON 载荷 |
| payloadTag | 16 B | 载荷 GCM 标签 |

`payloadLength` 不含末尾 `payloadTag`。解析器必须验证：文件总长度恰好等于上述固定长度加 `payloadLength`；所有保留位为零；KDF 参数处于客户端允许范围；没有尾随字节。

### 7.3 AAD 规则

为了防止攻击者替换算法标识、KDF 参数或长度字段：

- 包装 Export Key 时，AAD 为从 `magic` 到 `passwordSalt` 的所有字节（含 `passwordSalt`）；
- 加密载荷时，AAD 为从 `magic` 到 `payloadLength` 的所有字节，包含已写入的 `wrappedExportKey` 与 `wrapTag`；
- 三端必须使用相同的字节串，不得改用 JSON、结构体内存布局或平台默认编码。

### 7.4 导出流程

1. 用户在已解锁状态下发起导出，并设置独立导出密码；不得默认复用主密码，也不得把主密码预填入界面。
2. 校验导出密码强度并要求二次确认；生成随机 32 字节 Export Key 与随机盐、nonce。
3. 从数据库读取完整的记录、墓碑、冲突元数据和格式版本，生成第 8 节所述载荷。
4. 先以导出 KEK 包装 Export Key，再用 Export Key 加密载荷；按第 7.2 节写入临时文件。
5. 关闭并 fsync 临时文件，验证可重新解密后，再原子替换目标文件。失败时不覆盖已有文件。

### 7.5 导入流程

1. 在读取全部文件前，仅解析固定长度头部并执行尺寸、版本、参数上限检查。
2. 用户输入导出密码，派生导出 KEK，认证解密 Export Key；失败即终止。
3. 认证解密载荷；只有认证成功后才进行 JSON 解析。
4. 验证载荷 schema、UUID、时间、字段长度、记录数、重复 ID 和逻辑约束。
5. 在单个数据库事务中执行合并；成功后原子持久化，失败则回滚且不改变现有库。
6. 展示导入摘要：新增、相同跳过、冲突待处理、墓碑、无效拒绝的数量。不得在认证失败时泄露任何导入内容。

## 8. 加密载荷与数据模型

载荷使用 UTF-8 JSON，字段名固定为 ASCII。v1 不要求 JSON 字节的跨端确定性，因为 JSON 整体已被加密且仅在解密后解析；但所有时间均使用 UTC Unix 毫秒，所有 UUID 使用小写 RFC 4122 文本形式。

```json
{
  "schemaVersion": 1,
  "exportId": "uuid",
  "sourceVaultId": "uuid",
  "createdAtMs": 1760000000000,
  "records": [
    {
      "id": "uuid",
      "type": "login",
      "title": "示例网站",
      "username": "name@example.com",
      "password": "secret",
      "url": "https://example.com",
      "category": "工作",
      "note": "备注",
      "createdAtMs": 1760000000000,
      "updatedAtMs": 1760000000000,
      "revision": 4,
      "deviceId": "uuid"
    }
  ],
  "tombstones": [
    {
      "recordId": "uuid",
      "deletedAtMs": 1760000000000,
      "revision": 5,
      "deviceId": "uuid"
    }
  ],
  "conflictGroups": [
    {
      "groupId": "uuid",
      "recordId": "uuid",
      "state": "pending",
      "createdAtMs": 1760000000000,
      "variants": [
        { "kind": "record", "sourceVaultId": "uuid", "record": { "...": "完整 login 记录" } },
        { "kind": "tombstone", "sourceVaultId": "uuid", "tombstone": { "...": "完整墓碑记录" } }
      ]
    }
  ]
}
```

`conflictGroups` 仅包含未解决冲突；每组至少两个、最多十个变体。`kind` 只能为 `record` 或 `tombstone`，并且只能出现对应对象，不能同时出现。冲突解决后，组从该数组移除，并以正常记录或墓碑形式保存结果。

字段约束：`title` 1–256 字符；`username`、`url`、`category` 各不超过 2,048 字符；`password` 不超过 4,096 字符；`note` 不超过 16,384 字符。所有文本按 NFC 规范化，拒绝 NUL 字符和无效 UTF-8。密码字段仅在已解锁应用进程的必要内存中出现，磁盘、备份和导出文件中均只能出现加密后的字节。

## 9. 导入合并与冲突处理

### 9.1 基本规则

`updatedAtMs` 仅用于展示，不用于静默覆盖。导入算法以记录 ID 和内容摘要判断：

| 情形 | 操作 |
|---|---|
| 本地不存在该 ID，且没有对应墓碑 | 新增记录。 |
| 本地与导入内容相同 | 跳过，计为“已存在”。 |
| 同一 ID 内容不同 | 创建冲突组，保留本地版和导入版。 |
| 导入墓碑与本地记录冲突 | 创建冲突组，显示“删除 / 保留”的明确选择。 |
| 两边墓碑等价 | 跳过。 |

内容摘要使用 SHA-256 计算经固定字段顺序、UTF-8 NFC 编码的逻辑记录表示，仅用于库内相等判断，不单独作为安全认证机制。

### 9.2 用户处理冲突

冲突中心必须展示记录标题、来源设备、修改时间和关键字段差异，但默认遮蔽密码。用户可：

- 保留本地版本；
- 保留导入版本；
- 手动编辑为新版本；
- 对“删除/保留”冲突明确选择删除或恢复。

解决冲突会写入新的 `revision`，并保留审计元数据（冲突组 ID、解决时间、解决来源）；不保存旧密码明文的长期历史。未处理冲突必须在主界面显示计数，但不阻止用户使用其他记录。

### 9.3 墓碑保留

墓碑必须随每次导出携带，默认永久保留。由于不存在中心服务，客户端无法可靠知道所有旧快照是否已经过期；任何自动清理墓碑都可能导致删除的记录被旧文件恢复。若未来提供“压缩历史”功能，必须告知用户其风险并要求创建完整新备份。

## 10. 应用锁定与敏感数据处理

### 10.1 解锁、锁定和失败尝试

- 首次启动必须设置主密码，至少 12 字符；允许长口令，不强制复杂字符规则。
- 应用启动、从锁定态恢复、闲置 3/5/10 分钟超时后均要求解锁。
- 进入后台即遮蔽敏感界面；Android 应使用 `FLAG_SECURE` 降低截屏和最近任务泄露风险。Windows/macOS 尽力遮蔽窗口，但不承诺绝对禁止系统级录屏。
- 单次运行中连续失败 5 次后进入 30 秒冷却；之后按 30、60、120、300 秒递增，上限 5 分钟。重启会重置此体验级限制，因此它不能替代 Argon2id 对离线爆破的防护。
- 锁定时关闭数据库连接，清空 UI 缓存与搜索索引，尽可能覆盖可控的敏感缓冲区并释放 Vault Key、KEK、Export Key 引用。受托管运行时和操作系统虚拟内存机制限制，不宣称能够彻底擦除所有内存副本。

### 10.2 剪贴板

复制密码后默认 30 秒尝试清除剪贴板。清除前必须确认剪贴板仍是本应用写入的同一内容/版本；若用户已复制其他内容则不得覆盖。界面应提示复制倒计时，并允许用户立即清除。

### 10.3 生物识别

生物识别是可选的便利功能，不替代主密码：

- 首次启用必须在已用主密码解锁后进行；
- 仅允许系统安全硬件/Keychain/Keystore 保护用于解开 Vault Key 包装材料的密钥；
- 用户可随时关闭，关闭后删除平台凭据；
- 主密码修改后必须重新配置生物识别；
- 不支持生物识别的平台继续使用主密码。

## 11. 三端实现约束

| 平台 | UI 与存储 | 加密与安全注意事项 |
|---|---|---|
| Windows | WinUI 3（首选）+ SQLite/SQLCipher，存于 `%LocalAppData%` 私有目录，MSIX 打包。 | 使用 .NET 受支持的 AES-GCM 与安全随机源；Argon2id 使用经过审计且版本锁定的库。 |
| macOS | SwiftUI + SQLite/SQLCipher，存于 `Application Support`。 | CryptoKit 可提供 AES-GCM，但不直接提供 PBKDF2/Argon2id；使用经过审计的 Argon2id 实现。DMG 安装本身不等同于 App Sandbox。 |
| Android | Jetpack Compose + Room/SQLCipher，存于应用私有目录。 | 使用 `SecureRandom` 与受支持的 AES-GCM；不要把 Jetpack Security 当作跨端文件协议实现。关闭或排除自动备份，并启用 `FLAG_SECURE`。 |

三端不得各自“理解”协议。应维护一个语言无关的协议包，包含本规范、样例文件、明文输入、预期二进制十六进制、预期认证失败结果和版本升级案例。任何改动必须先更新协议版本和测试向量。

## 12. 交互与错误处理

- 所有导出、导入、改主密码、冲突处理均在完成后显示可理解的结果摘要。
- 对外错误文案不包含密钥、路径、数据库 SQL、堆栈、导出密码或原始字段内容。
- 导入前展示文件格式版本和数据量，不在认证完成前展示标题、账号或其他敏感内容。
- 不支持的未来格式版本提示“请升级应用”，旧版本应用不得尝试猜测解析新格式。
- 数据库或文件损坏时，优先保留原文件，提供“创建诊断副本（仍为密文）”与“从 `.pwdlock` 恢复”的路径，禁止自动覆盖损坏数据。

## 13. 测试与验收标准

### 13.1 必须自动化的互操作测试

1. 每个平台导出的 v1 文件必须能被另外两个平台导入；
2. 相同密码、盐、nonce、Export Key 和载荷输入时，三端必须生成完全一致的预期密文和认证标签测试向量；
3. 错误密码、错误标签、被改动的头部、截断、尾随字节、超长长度、未知版本和非法 JSON 必须安全拒绝；
4. 修改主密码后，旧主密码无法解锁、新主密码能解锁，数据库记录不变；
5. 冲突、删除/保留冲突、相同记录重复导入和墓碑导入均符合第 9 节；
6. 应用在导入、导出、改主密码的任意写入阶段被强制终止后，重启仍可打开旧库或完整新库，不能出现半写入库；
7. 自动锁定、后台遮蔽、剪贴板不误清除新内容等行为在各端手工验证。

### 13.2 发布门槛

- 加密协议和三端实现完成独立安全审查；
- 无测试向量不发布跨端导入导出；
- 无恢复演练不发布数据库迁移或格式升级；
- 依赖库版本、许可证、CVE 监控策略已记录；
- 隐私声明准确描述“不上传数据”与本节中的安全边界，禁止绝对化宣传语。

## 14. 分阶段交付

### 阶段 1：单端安全 MVP

完成 Windows 端主密码、信封加密、SQLCipher、登录项管理、自动锁定和本地备份恢复。先验证密钥生命周期、断电恢复与用户交互，不引入跨端复杂度。

### 阶段 2：协议与互操作

固化 `.pwdlock` v1 规范和测试向量；实现 Windows ↔ macOS ↔ Android 的导入导出矩阵；完成异常文件和冲突组处理。

### 阶段 3：体验增强

加入密码生成器、弱密码检查、分类搜索、生物识别、冲突中心和可验证备份提醒。

### 阶段 4：独立审计与发布

进行外部安全审查、发布签名与更新机制审查、迁移演练和用户文档完善。任何安全审查发现都优先于功能扩展处理。

## 15. 后续扩展原则

若未来加入 TOTP、附件、浏览器填充、更多字段或其他导出格式，必须通过新的 `schemaVersion` 或新的文件格式版本兼容演进。新增功能不得降低已有 KDF 参数、复用 AES-GCM nonce，或以“兼容性”为由导出明文密码。

本规格的核心判断是：离线架构和三端原生实现可以成立，但安全性取决于可审计的密钥管理、严格的协议实现、可验证的跨端测试与诚实的威胁模型，而不取决于自定义文件后缀或“其他软件看不懂”的表象。
